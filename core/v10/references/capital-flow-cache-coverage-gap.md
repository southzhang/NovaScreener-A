# capital_flow.json 缓存覆盖范围问题

## 问题描述

`capital_flow.json` 缓存只包含**固定持仓股/关注股**的资金面数据，而V10每日扫描产生的候选股是**动态变化**的。两者交集几乎为空。

## 实测数据（2026-05-19）

**缓存中的股票**（3只）：
- 600936 北投科技
- 300912 凯龙高科  
- 300065 海兰信

**当日V10候选股**（12只）：
- 600640 国脉文化、603311 金海高科、603999 读者传媒、605318 法狮龙、000685 中山公用、002927 泰永长征、300163 先锋新材、300473 德尔股份、300603 立昂技术、300329 海伦钢琴、002729 好利科技、600831 广电网络

**交集**：空（0只重合）

## 根因

`capital_flow.json` 的刷新任务（job `b00d3fde7beb`，每30分钟）是用固定的持仓/白名单股票作为查询参数，而不是从V10观察池动态读取。V10候选股每天不同，而缓存查询参数是静态的。

## 影响

- 尾盘选股/盘中选股任务的"资金面三维度验证"环节无法依赖缓存
- 候选股资金面数据要么走MCP实时查（慢，消耗额度），要么走push2批量替代（需要API可用）
- 如果MCP在cron上下文中不可用且push2被VPN拦截，资金面验证必须降级

## 缓解方案

1. **扩展缓存股票池**：`capital_flow_cache.py` 改为同时读取 `v10_watchlist.json` 中的候选股，不仅限于持仓/白名单
2. **实时降级路径**：选股任务中直接用 push2 批量API（fltt=2, fields=f62,f184）查当前候选股的资金面，不依赖缓存
3. **无数据时降级**：资金面数据不可得时，标注"待验证"，不阻断推荐（但建议降级优先考虑）

## 铁律

- 选股任务不能假设 `capital_flow.json` 里有当前候选股的数据
- 资金面验证层必须有不依赖缓存的备用查询方案
- 如果所有资金面数据源均不可用，标注"✅ 资金面：缓存miss，数据源不可用"而非静默忽略
