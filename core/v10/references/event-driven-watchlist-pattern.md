# 事件驱动盯盘模式：向已有任务追加临时标的

## 场景

当出现行业级事件（长鑫存储业绩爆发、政策利好等）需要临时监控一批产业链标的时，不应新建专用盯盘任务，而是在现有实盘盯盘任务中追加监控标的。

## 为什么追加而非新建

- 新建任务增加调度成本，且可能被遗忘清理
- 事件热度消退后，需要记得下掉临时盯盘
- 现有实盘盯盘任务已有完整的数据架构（腾讯行情→iFinD MCP→推送规则）

## 操作步骤

### 1. 识别目标盯盘任务

```bash
cronjob list → 找到 "实盘盯盘" 类Agent任务（非脚本任务）
# 当前: 26daefd24af7 "实盘盯盘-持仓监控+iFinD分析"
```

### 2. 梳理产业链标的

用 iFinD MCP 搜索 + web_search 产业链分析，按受益弹性排序：
- 直投/核心供应商（最优先）
- 设备/材料（次优先）
- 间接概念股（低优先）

### 3. 用盘前数据验证

在集合竞价前跑一次行情确认标的的当前状态（价格、近5日涨跌、是否有异动）。

### 4. 更新 cron prompt

追加新的监控标的代码列表，并标注为"今日特别关注"。同时设置推送规则：
- 异常（涨跌幅>±3%、触及止损）→ 立即推送
- 正常 → 静默

## 盯盘任务 prompt 结构模板

```
你是A股实盘盯盘助手，执行持仓监控+基本面分析+{事件主题}盯盘。

## 📊 盯盘范围神明

### 实盘持仓
{code1} {name1}, {code2} {name2}, ...

### {事件主题}产业链重点监控（今日特别关注）
- **{code} {name}** — {核心逻辑}
- **{code} {name}** — {核心逻辑}

## 推送规则
- 持仓异常 → 立即推送摘要
- {事件主题}标的异动 → 推送并标注主题
- 全部正常 → 静默
```

## 实操案例：长鑫存储业绩爆发（2026-05-18）

标的梳理：
| 层级 | 标的 | 代码 | 核心逻辑 |
|------|------|------|----------|
| 直投 | 兆易创新 | 603986 | 唯一A股直接持股1.8%，关联采购57亿 |
| 封测 | 通富微电 | 002156 | HBM封装核心，长鑫第一大客户(45%订单) |
| 设备 | 北方华创 | 002371 | 刻蚀/薄膜沉积设备龙头 |
| 材料 | 雅克科技 | 002409 | 前驱体/电子特气 |

追加到 26daefd24af7 实盘盯盘任务，9:10 开始首轮扫描。

## 清理

事件热度消退后，从 cron prompt 中移除临时标的，恢复为仅持仓盯盘。

⚠️ **铁律**：临时标的仅监控异动，不自动交易。用户自行判断是否入场。

## ⚠️ 行情获取脚本兜底方案

cron prompt 中可能引用 `python3 ~/.hermes/scripts/tencent_realtime.py`，但该脚本**不存在**。实际行情获取方式：

**兜底（已验证可靠）：**
```bash
curl -s "https://qt.gtimg.cn/q=sh600936,sz300912,sz300065,..." | iconv -f gbk -t utf-8
```

解析腾讯行情 `parts` 数组：
- parts[1]=名称, parts[3]=现价, parts[4]=昨收, parts[5]=今开
- parts[32]=涨跌幅(%), parts[33]=最高, parts[34]=最低
- parts[37]=成交额(万元)

**铁律**：不要依赖 `tencent_realtime.py`——直接用 curl + 腾讯行情，无需封装脚本。如果 cron prompt 中有该引用且任务失败，应将 prompt 中的脚本调用替换为 curl 命令。
